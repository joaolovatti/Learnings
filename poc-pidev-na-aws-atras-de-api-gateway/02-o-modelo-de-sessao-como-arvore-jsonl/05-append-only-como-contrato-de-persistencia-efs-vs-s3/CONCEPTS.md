## 1. O ciclo de vida do arquivo de sessão na abertura — como o harness lê o arquivo inteiro para reconstruir o índice in-memory e depois abre em append; o que "continuar uma sessão" significa em termos de operações de arquivo

Continuar uma sessão pi.dev não é a mesma operação que retomar um processo pausado — é uma leitura completa do arquivo seguida de uma escrita incremental. O harness nunca mantém o arquivo aberto entre invocações; quando o usuário executa `pi --continue` ou o SDK chama `SessionManager.open()`, o arquivo inteiro é lido do início ao fim antes de qualquer nova linha ser escrita.

Durante essa leitura inicial, o `SessionManager` constrói um mapa `id → entrada` em memória. Cada linha do JSONL é uma entrada com um campo `id` e, exceto o `SessionHeader` da primeira linha, um campo `parentId`. Lendo sequencialmente, o harness reconstrói o grafo dirigido acíclico que representa a árvore de sessão — tronco, branches, compactions — exatamente como foi detalhado no subcapítulo `02-a-arvore-de-sessao-dag-trunk-e-branches`. Esse mapa em memória é o único mecanismo de navegação disponível: não há arquivo de índice separado, não há banco de dados auxiliar. Se o arquivo de sessão desaparecer, a árvore some com ele.

O processo pode incluir migração de versão antes de finalizar o índice: sessões no formato v1 ou v2 são convertidas para o esquema v3 (com campo `id` de 8 hex chars e `parentId`) nessa fase de leitura, de forma transparente. Somente depois que o índice in-memory está completo e consistente é que o harness avança para o modo de escrita.

Na fase de escrita, o comportamento muda completamente: o arquivo é aberto em modo append e cada novo `appendEntry()` acrescenta uma única linha ao final sem tocar nas anteriores. A chamada atualiza simultaneamente o mapa em memória (`id → entrada`) e persiste no disco. Não há "flush" periódico nem buffer intermediário para entradas individuais — a linha vai para o arquivo no momento da chamada. Isso significa que uma sessão com 200 entradas exige uma leitura de 200 linhas na abertura e, a partir daí, escrita de 1 linha por turno.

A implicação direta para persistência externa é que "continuar uma sessão" exige que **o arquivo completo esteja disponível e íntegro antes do primeiro turno**. Um sistema de storage que entregue um arquivo truncado, com linhas faltando no meio, ou com o `SessionHeader` corrompido vai produzir um índice in-memory parcial ou inválido — o harness não tem como saber que o arquivo está incompleto, pois lê até o EOF e assume que tudo que leu é verdade. Esse comportamento é o ponto de partida para entender por que EFS e S3 têm implicações radicalmente diferentes: EFS entrega um arquivo de sistema de arquivos com semântica POSIX completa; S3 entrega um objeto imutável que precisa ser reconstruído na totalidade antes de qualquer nova escrita — terreno dos conceitos `## 4.` e `## 6.` deste subcapítulo.

## 2. O_APPEND no POSIX e por que EFS o suporta nativamente — a semântica de `open(O_APPEND)` como garantia de offset atômico ao final do arquivo, e por que EFS (sistema de arquivos POSIX sobre NFS4) herda esse comportamento

O problema que `O_APPEND` resolve pode ser visto pela negativa: se o harness abrisse o arquivo de sessão sem essa flag e tentasse posicionar-se no final com `lseek()` antes de cada `write()`, haveria uma janela de tempo — entre o `lseek` e o `write` — em que outro processo poderia estender o arquivo. O offset calculado pelo `lseek` estaria desatualizado quando o `write` executasse, e a nova linha seria escrita sobre dados existentes, corrompendo silenciosamente o JSONL. É exatamente essa condição de corrida que torna o padrão lseek-then-write inseguro para escritas concorrentes.

`O_APPEND` colapsa as duas operações em uma só: ao chamar `write()` em um descritor aberto com essa flag, o kernel garante que o offset seja posicionado no EOF **e** que a escrita ocorra como um passo atômico único, sem nenhuma modificação de terceiros podendo se intercalar entre o posicionamento e a gravação. O que o POSIX especifica é preciso: "the file offset shall be set to the end of the file prior to each write and no intervening file modification operation shall occur between changing the file offset and the write operation." Essa atomicidade não significa que duas escritas simultâneas de múltiplos kilobytes não possam ter seus bytes físicos intercalados — o limite para dados não entrelaçados é `PIPE_BUF` (tipicamente 4 KB no Linux) — mas para o caso do pi.dev, onde cada linha JSONL é da ordem de algumas centenas de bytes, essa garantia é suficiente: cada `appendEntry()` produz uma linha contígua e não embaralhada ao final do arquivo.

A relevância para a POC fica clara quando se considera o capítulo 8, que trata de EFS multi-tenant com access points. O harness pi.dev é um processo que abre o arquivo em modo append no início de cada turno e o fecha ao fim; em um ambiente onde múltiplos containers Fargate poderiam estar acessando sessões diferentes, mas montando o mesmo sistema de arquivos EFS, o kernel de cada instância gerencia descritores independentes no mesmo namespace NFS. Cada processo vê as garantias locais de `O_APPEND` porque o EFS suporta semântica POSIX completa — incluindo `open()`, `write()`, `lseek()` e file locking NFSv4 — como sistema de arquivos de rede.

O elo entre `O_APPEND` local e EFS passa pelo NFS4. O EFS usa NFSv4.1 como protocolo de transporte e expõe ao kernel do cliente exatamente a interface de arquivos POSIX que um processo em Linux espera. Quando o processo chama `open(path, O_WRONLY | O_APPEND)`, o kernel Linux traduz isso para operações NFS que preservam a semântica de append: internamente, o servidor EFS trata a abertura com o modo `OPEN4_SHARE_ACCESS_WRITE` e as escritas subsequentes são processadas sequencialmente pelo servidor com o offset posicionado no final do arquivo. O resultado visível para o processo é idêntico ao de um sistema de arquivos local ext4: cada linha escrita aterra atomicamente ao final, sem colisão com outras escritas do mesmo host. A ressalva importante — que o `## 3.` deste subcapítulo desenvolverá — é que "do mesmo host" é a chave: a atomicidade que EFS fornece é garantida dentro do contexto de um único ponto de montagem, mas a propagação de escritas entre Availability Zones diferentes introduz latência que `O_APPEND` por si só não resolve.

Para o pi.dev especificamente, essa combinação é bem-casada: o harness nunca mantém dois descritores simultâneos no mesmo arquivo de sessão (cada invocação é serial — leitura completa, depois escritas de um turno por vez), de modo que o EFS não precisa arbitrar escritas concorrentes no mesmo arquivo. O `O_APPEND` funciona aqui como seguro: mesmo que um bug de implementação ou um restart abrupto deixasse dois processos apontando para o mesmo arquivo de sessão, a flag previne a sobreposição silenciosa. Sem ela, a corrupção seria invisível até a próxima abertura, quando o mapa `id → entrada` construído na fase de leitura produziria entradas com `parentId` inconsistente — o mesmo tipo de falha tratada no subcapítulo `06-operacoes-proibidas-e-restricoes-de-integridade`.

## 3. A tensão de latência multi-AZ no EFS — por que EFS é natural para sessões pi.dev mas a propagação de escritas entre Availability Zones tem latência mensurável, e quando isso vira problema real

O `## 2.` estabeleceu que `O_APPEND` garante atomicidade ao nível de escrita individual. Mas atomicidade local não é o mesmo que visibilidade global — e em um sistema de arquivos distribuído que replica dados entre Availability Zones, essas duas propriedades se separam no tempo.

O modelo de consistência do EFS é **close-to-open**: uma escrita torna-se visível para outros clientes no sistema de arquivos quando o processo que escreveu fecha o arquivo. Antes do `close()`, os dados podem estar em buffer no cliente NFS — visíveis para o processo que escreveu, mas não necessariamente para processos rodando em outras máquinas (ou em outras AZs). Para Regional file systems, a garantia do EFS é precisa: "write operations are durably stored across Availability Zones when an application performs a synchronous write operation (for example, using `O_DIRECT` or `fsync`) or when an application closes a file." Isso significa que o commit cross-AZ é acionado pelo `close()` ou por um `fsync()` explícito — não por cada `write()` intermediário.

Para o pi.dev, esse modelo é bem-casado por acidente de design. O harness nunca mantém o arquivo de sessão aberto entre invocações: cada turno consiste em (1) `open()` para leitura do arquivo inteiro, (2) construção do índice in-memory, (3) `open(O_APPEND)` para escrita das novas entradas, e (4) `close()` ao encerrar a invocação. O `close()` ao final do turno é exatamente o gatilho que o EFS usa para confirmar a escrita em todas as AZs. Portanto, quando o próximo turno começa — possivelmente em um container Fargate diferente, em uma AZ diferente — a abertura de leitura do arquivo vai encontrar os dados do turno anterior já duravelmente propagados. O ciclo de vida serial das invocações do harness transforma a semântica close-to-open de "possível armadilha NFS" para "garantia estrutural".

A latência mensurável entra quando se examina o que acontece entre o `close()` de um turno e o `open()` do turno seguinte rodando em AZ diferente. O EFS Standard (SSD) tem latência de primeiro byte de aproximadamente 2,7 ms para escritas em condições normais de acesso local. Quando a task Fargate que escreveu está em AZ-a e a task que vai ler está em AZ-b, o commit cross-AZ precisa cruzar o backbone interno da AWS — o EFS documenta que replica por três ou mais AZs, mas não especifica o tempo exato de propagação após o `close()`. Na prática, o overhead é da ordem de dezenas de milissegundos para arquivos pequenos, não de segundos. Para sessões do pi.dev, onde cada turno pode levar segundos a minutos, essa janela é irrelevante.

O cenário em que a latência vira problema real é o de múltiplas invocações rápidas simultâneas tentando usar o mesmo arquivo de sessão a partir de AZs diferentes — por exemplo, se um cliente disparar dois turnos antes do primeiro encerrar. Nesse caso, a task que começou o segundo turno pode abrir o arquivo para leitura antes que o `close()` da primeira task tenha propagado, produzindo um índice in-memory incompleto: a fase de leitura inicial vai ler o arquivo até o EOF do que está disponível naquele instante, e o turno anterior simplesmente não vai aparecer no grafo. Essa situação não é diferente de um bug de locking de aplicação — o EFS entrega o que o protocolo NFS promete; quem falhou foi a camada acima que não serializa turnos por sessão. Para a POC descrita no capítulo 6 (`06-api-gateway-na-frente-rotas-para-criar-sessao-enviar-turno-e-listar-sessoes`), a serialização natural das rotas HTTP é suficiente para evitar isso na prática.

Há um segundo ângulo de latência que não tem relação com consistência: a penalidade de rede cross-AZ no caminho NFS. O EFS expõe um mount target por AZ; quando a task Fargate está em AZ-a mas o mount target acessado está em AZ-b (por exemplo, por má configuração de subnets no serviço ECS), cada operação NFS — incluindo `stat()`, `open()` e cada `write()` de linha — atravessa o link cross-AZ, adicionando dezenas de milissegundos por chamada. Em casos documentados com workloads de arquivos pequenos sobre EFS, a diferença entre acesso same-AZ e cross-AZ foi de 4 segundos vs 30 segundos para operações de varredura de diretório. Para o pi.dev, a fase de leitura inicial que itera linha a linha sobre um arquivo de sessão grande seria a mais afetada. A solução é configuracional, não de código: garantir que as tasks Fargate sejam lançadas nas mesmas AZs que os mount targets EFS que usam. Isso está no território do capítulo 8 (`08-efs-multi-tenant-por-access-point-sessoes-pidev-sobreviventes`), que trata de access points por tenant.

## 4. Por que S3 não tem append nativo e os dois patterns de contorno — imutabilidade do objeto em buckets padrão, reescrita completa vs acumulação + flush periódico, e o trade-off de cada abordagem

O contraste com o EFS começa pelo modelo de dados. O EFS é um sistema de arquivos: objetos endereçados por caminho, mutáveis, com operações de escrita posicional que o kernel traduz para chamadas NFS. O S3 é um armazenamento de objetos: cada chave aponta para um blob imutável — substituível como unidade, mas não editável em partes. Não há conceito de "abrir em append", "posicionar cursor" ou "escrever nos últimos N bytes". Quando você chama `PutObject`, o objeto antigo é descartado e substituído pelo novo na íntegra. Isso não é uma limitação de API; é a fundação do modelo de consistência do S3, que otimiza para leitura barata e altamente paralela de objetos inteiros, não para escritas incrementais.

A consequência direta para o pi.dev: cada turno de sessão acrescenta pelo menos uma linha ao arquivo JSONL. No EFS, isso é uma chamada `write()` de alguns bytes. No S3 padrão, isso exigiria baixar o objeto inteiro, adicionar a linha no buffer local, e subir o objeto inteiro de volta — custando em latência e em requisições PATTs proporcionais ao tamanho acumulado da sessão, não ao tamanho do turno. Uma sessão com 150 entradas de média de 300 bytes cada (≈ 45 KB) geraria um ciclo download + upload de 45 KB a cada turno, mesmo que a nova entrada seja de 250 bytes. O custo cresce linearmente com o tamanho da sessão.

Existe uma exceção recente que vale nomear antes de descartá-la: a `S3 Express One Zone` (directory buckets) introduziu append nativo via `PutObject` com o header `x-amz-write-offset-bytes`. A operação é genuinamente incremental — sem reescrita do objeto — e suporta até 10.000 appends por objeto. O problema é que S3 Express One Zone é uma classe de armazenamento diferente, disponível apenas em regiões e AZs específicas, com modelo de preço distinto do S3 padrão e sem a mesma durabilidade multi-AZ (11 noves no S3 Standard vs. armazenamento em AZ única no Express One Zone). Para a POC descrita neste livro, que usa S3 Standard como alternativa ao EFS, `S3 Express One Zone` não é um substituto direto — é uma terceira opção com trade-offs próprios que o capítulo 9 (`09-sessionmanager-customizado-backed-por-s3`) pode ou não considerar.

No S3 Standard, os dois patterns de contorno são: **reescrita completa** e **acumulação + flush periódico**.

A **reescrita completa** é o approach mais simples: a cada novo turno, o `SessionManager` baixa o objeto atual, adiciona as novas entradas no buffer em memória, e faz `PutObject` com o conteúdo inteiro. O código é trivial — uma linha de `GetObject` + concatenação + `PutObject` — e o resultado é sempre um objeto íntegro e legível pelo harness sem nenhuma lógica de reconstrução. O problema é duplo: custo de I/O cresce com o tamanho da sessão (cada turno acessa todo o histórico, não só a delta), e a janela de race condition é proporcional ao tempo de download + processamento + upload. Se dois turnos começarem antes do primeiro completar o `PutObject`, o último a escrever vence e o dado do outro desaparece — o chamado last-write-wins, que para sessões pi.dev significa perda silenciosa de entradas do índice in-memory do próximo turno. Isso é tratado no `## 6.` deste subcapítulo.

A **acumulação + flush periódico** parte de uma constatação diferente: se o harness pi.dev for o único processo escrevendo na sessão em um dado momento (modelo serial por sessão que a POC implementa), então não é necessário subir cada turno individualmente ao S3. Em vez disso, o `SessionManager` mantém as novas entradas em buffer local — disco temporário do container, `/tmp` em Lambda, EFS de trânsito — e periodicamente ou ao encerrar a sessão consolida tudo em um único `PutObject`. O benefício é óbvio: um único upload ao final em vez de N uploads durante o uso. O trade-off é a janela de perda: se o container morrer antes do flush, todas as entradas acumuladas desde o último `PutObject` são perdidas. O tamanho dessa janela é uma decisão de negócio — flush a cada turno recupera a simplicidade da reescrita completa (com o custo de I/O correspondente), flush a cada 10 turnos reduz custo mas aumenta exposição a perda de dados. Não há como ter janela zero e custo constante ao mesmo tempo no S3 Standard com sessões que crescem.

Uma variante da acumulação que aparece em contextos de log distribuído é o pattern de **partes nomeadas**: em vez de um único objeto `session-<id>.jsonl`, o `SessionManager` escreve `session-<id>/part-0001.jsonl`, `session-<id>/part-0002.jsonl`, e assim por diante — cada `PutObject` carrega apenas as novas entradas desde o último flush. Para ler a sessão completa, o código lista o prefixo, ordena as partes por nome e concatena em memória antes de passar para o harness. O custo de escrita é constante por turno (só as novas linhas). O custo de leitura cresce com o número de partes (N requisições `GetObject` em vez de 1). A desvantagem crítica para o pi.dev é que o harness espera receber um único stream de linhas na ordem de inserção original — concatenar partes em ordem é necessário, mas a ordenação correta das partes não é trivial se houver concorrência ou retransmissão, e a semântica de `id`/`parentId` não tolera reordenação. Esse problema é o tema do `## 5.`.

Nenhum dos dois patterns elimina o problema de fundo — S3 Standard não foi projetado para escritas incrementais em objetos que crescem linha a linha. O que muda entre eles é onde o custo vai: na reescrita completa, o custo é de bandwidth e latência crescente por turno; na acumulação, o custo é de lógica de reconstrução na leitura e de exposição a perda de dados entre flushes. Para uma POC cujo objetivo é demonstrabilidade (não performance de produção), a reescrita completa é o ponto de partida mais defensável — menos estados inconsistentes possíveis, mais fácil de debugar — e é por isso que o capítulo 9 tende a começar por ali antes de introduzir otimizações.

## 5. A ordem das linhas JSONL não pode ser ignorada ao serializar para S3 — por que a semântica de árvore (id/parentId) torna a ordem de append parte da estrutura lógica, e o que acontece se o merge de partes for feito fora de ordem

O `## 4.` apresentou o pattern de partes nomeadas como a variante da acumulação que distribui o custo de escrita — em vez de um único `session-<id>.jsonl` que cresce e precisa ser reescrito inteiro, o `SessionManager` produz `session-<id>/part-0001.jsonl`, `part-0002.jsonl`, e assim por diante, e a leitura da sessão concatena as partes em ordem. A questão que esse pattern deixou em aberto é: o que exatamente significa "em ordem", e qual a consequência de errar?

A resposta exige entender como o `SessionManager` constrói o índice in-memory durante a fase de abertura descrita no `## 1.`. A implementação lê todas as linhas do arquivo JSONL para um array e percorre esse array uma única vez para popular o mapa `byId: Map<id, entry>`. Ao terminar essa passagem, o último `id` encontrado é atribuído ao `leafId` — o ponteiro que indica qual é a folha ativa da árvore, ou seja, o nó a partir do qual o próximo turno vai crescer. Não há resolução posterior de "qual é o último nó por timestamp" ou "qual é o nó com maior profundidade de árvore": o critério é posicional, determinado por qual entrada apareceu por último no stream de bytes.

Isso transforma a posição da última linha em informação estrutural, não apenas em metadado de criação. Em uma sessão linear simples (trunk sem branches), todas as entradas formam uma cadeia única onde cada `parentId` aponta para o `id` da linha anterior. O `leafId` correto é o nó do turno mais recente, que é também o último a ter sido escrito em append — e que, portanto, aparece na última linha do arquivo se o arquivo foi gerado pelo harness sem edição. Se o merge das partes invertesse a ordem de duas partes quaisquer, o `leafId` que o harness registraria seria o último nó da parte que ficou no final do arquivo, que agora corresponderia a um turno anterior — e o próximo `appendEntry()` cresceria a partir desse nó, não do mais recente, criando um branch indesejado sem que o usuário tenha pedido fork.

O efeito não é uma exceção nem um crash visível: o `SessionManager` aceita silenciosamente o estado que recebe. O mapa `byId` fica correto — todas as entradas estão lá, com seus `id` e `parentId` legítimos. O que está errado é o `leafId`. A sessão "continua" de um ponto que o usuário não escolheu, e o novo turno passa a ser filho de um nó do meio do histórico, não da ponta. O próximo `buildSessionContext()` vai traçar o caminho de volta à raiz a partir desse `leafId` deslocado, produzindo um contexto de LLM truncado — sem as entradas que apareceram depois desse nó na ordem original. O modelo recebe uma visão incompleta da conversa e responde como se parte do histórico não tivesse acontecido.

O caso de branch complica a análise, mas aprofunda o problema. Quando o arquivo de sessão contém um fork — múltiplas folhas coexistindo, como `part-0003.jsonl` registrou a criação de um branch e `part-0004.jsonl` continuou o branch original — a ordem relativa das partes determina qual folha o harness vai selecionar como ativa. Se o merge inverter `part-0003` e `part-0004`, o `leafId` vai apontar para o nó correto do ponto de vista do arquivo original. Mas se o merge de uma variante do pattern produzir uma concatenação onde as entradas do branch aparecem intercaladas fora da sequência cronológica, o resultado pode ser um `leafId` que aponta para uma folha morta — um nó que tinha sido suplantado por seu irmão mais recente.

Há ainda um terceiro ângulo, específico ao multipart upload do S3, que o `## 4.` mencionou em passagem. Quando o `SessionManager` usa `CompleteMultipartUpload` para consolidar partes numa única chave S3, a API do S3 concatena as partes em ordem crescente de `PartNumber`, independentemente da ordem em que as partes foram carregadas. Se o código atribuir `PartNumber` 1 à `part-0002.jsonl` e `PartNumber` 2 à `part-0001.jsonl` — por um bug de mapeamento inverso ou por re-numeração durante retransmissão — o objeto final terá as partes na ordem errada. O S3 não valida o conteúdo, só o `PartNumber`; ele garante a ordem dos bytes de cada parte, não a correção semântica do JSONL resultante. O erro é silencioso até o próximo `SessionManager.open()`, e o diagnóstico requer listar as partes de um upload antigo para comparar os `PartNumber` com o conteúdo esperado.

A assimetria com o EFS é completa aqui. No EFS, cada `appendEntry()` chama `write()` com `O_APPEND`, o kernel posiciona o offset no EOF atomicamente e grava — a ordem das linhas no arquivo é a ordem de chegada das chamadas `write()`, e o harness nunca pode interferir nessa ordem porque ele mesmo produz as escritas em sequência serial. No S3 com partes, a ordem de concatenação é uma responsabilidade explícita do código que chama `CompleteMultipartUpload` — ela não é uma propriedade emergente do sistema de arquivos, é um contrato que o código tem de honrar ativamente. Essa diferença de onde vive a responsabilidade da ordem é o que torna o S3 mais frágil para sessões com branches: o EFS oferece a invariante de graça; o S3 exige que o código a construa e mantenha.

## 6. Read-after-write: EFS garante, S3 é fortemente consistente mas o race condition de reescrita ainda existe — a diferença entre consistência de leitura e atomicidade de escrita concorrente no contexto de sessões com múltiplos turnos rápidos

Os conceitos anteriores estabeleceram dois problemas distintos no S3: a falta de append nativo (`## 4.`) e a fragilidade da ordenação de partes (`## 5.`). O `## 6.` fecha o diagnóstico: mesmo com o S3 fortemente consistente desde dezembro de 2020, a consistência de leitura e a atomicidade de escrita concorrente são propriedades separadas — e confundi-las é o erro de modelagem mais comum ao desenhar um `SessionManager` sobre S3.

**Consistência de leitura** é a garantia de que uma leitura feita após uma escrita bem-sucedida vê o dado escrito — e não uma versão anterior ou um estado fantasma. O S3 Standard garante isso para todas as operações GET, PUT e LIST desde a atualização de 2020: um `GetObject` imediatamente após um `PutObject` bem-sucedido retorna obrigatoriamente o objeto recém-escrito. Essa garantia inclui sobrescritas — se você fez `PutObject` de um objeto que já existia, o próximo `GetObject` retorna a nova versão, nunca a antiga. Para a fase de abertura do `SessionManager` descrita no `## 1.` — que começa por ler o arquivo de sessão inteiro — isso significa que qualquer turno que tiver completado seu `PutObject` com sucesso será completamente visível na próxima leitura. Não há janela de "eventual consistency" nem necessidade de polling até o dado aparecer.

**Atomicidade de escrita concorrente** é uma propriedade diferente: garante que duas escritas simultâneas para a mesma chave não produzam um estado corrompido ou parcial. O S3 não oferece essa propriedade diretamente — ele usa semântica de **last-writer-wins**: quando dois `PutObject` chegam ao mesmo tempo para a mesma chave, ambos são aceitos, o servidor processa um depois do outro, e o último a ser confirmado vence. O objeto resultante é íntegro (não é um merge de bytes das duas requisições), mas o conteúdo do primeiro `PutObject` é descartado silenciosamente. O S3 não retorna erro para nenhum dos dois clientes; ambos recebem HTTP 200, e nenhum sabe que seu dado foi sobrescrito.

Para sessões pi.dev com o pattern de reescrita completa do `## 4.`, o race condition concreto funciona assim: o turno A baixa o objeto de sessão (`GetObject`), processa a resposta do LLM, e está prestes a subir o objeto atualizado. Enquanto isso, o turno B — disparado pelo usuário antes de A terminar — também baixou o mesmo objeto, processou sua resposta, e faz `PutObject` primeiro. Turno A então faz seu `PutObject` com o estado que baixou antes de B existir — sobrescrevendo o objeto que B acabou de escrever. O resultado é um arquivo de sessão que contém as entradas do turno A mas não as do turno B. O S3 não detectou nem sinalizou o conflito: do ponto de vista do protocolo, ambas as escritas foram bem-sucedidas. O `leafId` da próxima abertura vai apontar para o último nó do turno A, como se B nunca tivesse acontecido.

```
Turno A: GetObject(t=0) → PutObject(t=3)  [win]
Turno B: GetObject(t=1) → PutObject(t=2)  [overwritten by A at t=3]

Estado final: entradas de A presentes, entradas de B perdidas silenciosamente.
```

A diferença com o EFS é estrutural. No EFS, cada `appendEntry()` é uma chamada `write()` com `O_APPEND` — o kernel posiciona o offset no EOF atomicamente e grava. Não há janela entre "baixar o estado atual" e "escrever no final": o sistema de arquivos resolve o offset em tempo real, e duas escritas simultâneas produzem duas linhas no final do arquivo, uma depois da outra, sem conflito. O EFS não tem o problema de last-writer-wins porque nunca há um ciclo de get-modify-put — cada escrita é incremental por design.

O mecanismo que o S3 oferece para fechar essa lacuna são as **conditional writes** com header `If-Match` (disponíveis desde agosto de 2024). A ideia é a do compare-and-swap: o cliente lê o objeto, guarda o ETag retornado no `GetObject`, e ao fazer o `PutObject` inclui `If-Match: <etag>`. O S3 só aceita a escrita se o ETag atual do objeto ainda for o mesmo que o cliente viu. Se outra escrita aconteceu entretanto — mudando o ETag — o S3 retorna HTTP 412 Precondition Failed, e o cliente sabe que precisa re-ler o objeto, re-aplicar sua mudança, e tentar de novo. Isso transforma o last-writer-wins silencioso em conflito explícito e recuperável.

Para o `SessionManager` sobre S3, a diferença prática é entre duas arquiteturas:

| Abordagem | Protocolo de escrita | Comportamento em race |
|---|---|---|
| PutObject simples | GetObject → modify → PutObject | Último a escrever vence, sem erro |
| PutObject com If-Match | GetObject (guarda ETag) → modify → PutObject(If-Match: etag) | Conflito retorna 412, cliente re-tenta |

A abordagem com `If-Match` transforma o problema de "dados perdidos silenciosamente" para "retry explícito com re-leitura" — o que é muito mais defensável para sessões onde cada turno tem valor. O custo é uma lógica de retry no cliente e a possibilidade de contenção alta em sessões muito ativas (cada retry re-lê o objeto inteiro e re-aplica a modificação). Para a POC deste livro, onde o modelo serial de turnos da rota HTTP já serializa a maioria dos acessos na prática, a janela de race é pequena — mas o capítulo 9 (`09-sessionmanager-customizado-backed-por-s3`) é o lugar onde essa escolha se torna explícita em código.

O ponto final de comparação entre EFS e S3 neste subcapítulo é então: EFS resolve read-after-write e atomicidade de escrita concorrente como propriedades emergentes do sistema de arquivos POSIX — o harness não precisa saber que está em um ambiente distribuído. S3 resolve read-after-write via consistência forte nativa, mas delega a atomicidade de escrita concorrente para o código da aplicação — que precisa explicitamente usar `If-Match` para obter semântica equivalente. Nenhuma das duas soluções é errada; são trade-offs que o capítulo 9 precisa honrar em cada linha de código do `SessionManager`.
