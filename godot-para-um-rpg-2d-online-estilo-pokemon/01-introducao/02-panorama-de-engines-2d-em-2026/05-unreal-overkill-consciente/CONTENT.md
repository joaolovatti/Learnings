# Unreal — Overkill Consciente

![capa](cover.png)

GameMaker, Defold e Construct falharam na dimensão eliminatória 2 — cada uma por um caminho diferente, mas todas sem saída: rollback networking inadequado para um RPG servidor-autoritativo, Photon externo sem modelo para mundo persistente, WebRTC peer-to-peer sem autoridade real de servidor. Antes de fechar o panorama, há uma última candidata que merece menção — não porque passe no crivo, mas porque o tamanho da falha dela é instrucional e porque ignorá-la poderia parecer uma lacuna deliberada. Unreal Engine é a engine mais poderosa do mercado em 2026. É também, para este projeto, o caso mais claro de inadequação.

A Epic Games lançou o Unreal Engine em 1998 como engine de first-person shooters. Em 2015, com o UE4, abriu o código e tornou o acesso gratuito (com royalty de 5% sobre receita acima de $1 milhão). Em 2022 lançou o UE5 com Lumen, Nanite e Chaos Physics — tecnologias que redefinem o teto de fidelidade visual em jogos 3D de alto orçamento. Fortnite, The Matrix Awakens, Senua's Saga: Hellblade II e Black Myth: Wukong rodaram em Unreal. A engine é a referência do segmento AAA 3D.

Esse background explica tudo sobre a inadaptação ao projeto. Unreal nasceu, evoluiu e continua sendo investida como engine 3D de alto orçamento — os avanços do UE5 não apenas ignoram o 2D como ativamente encarecem o custo de operação. Instalar o Unreal Engine 5 exige entre 60 GB e 115 GB em disco dependendo dos componentes selecionados, mais 100 GB recomendados para projetos com assets. Para comparação, Godot 4 completo tem ~80 MB. Antes de escrever uma linha de código para o jogo, o desenvolvedor já gastou horas de download, gigabytes de SSD e teve que entender o Epic Games Launcher — não por ganho algum relacionado ao projeto, mas pelo overhead estrutural de uma ferramenta projetada para outra escala.

O suporte a 2D em Unreal existe via **Paper2D**, um plugin disponível desde o UE4. A história do Paper2D é reveladora: discussões no fórum oficial questionando se o desenvolvimento parou se repetem desde 2016. O plugin existe, é documentado, e há tutoriais de RPG top-down feitos com ele — mas os recursos que em Godot são de primeira classe (TileMapLayer com autotile, física integrada ao tilemap, AnimatedSprite2D com AnimationPlayer nativo) exigem configuração manual extensa ou gambiarra arquitetural no contexto do Paper2D. O editor do Unreal abre em modo 3D por padrão — perspectiva, eixos `X/Y/Z`, configurações de iluminação dinâmica. Trabalhar em top-down 2D requer configurar manualmente câmera ortogonal, desativar Lumen e Nanite (features cuja simples existência na interface de configuração é distração constante), e travar a projeção. O projeto inteiro opera num estado de "estou usando 3% desta engine".

Na dimensão de custo de aprendizado, Unreal é a mais custosa de todas as candidatas. O sistema de scripting oferece duas opções: **Blueprints** (programação visual por nós) ou **C++**. Blueprints são poderosos, mas à medida que a lógica de jogo cresce — máquinas de estado de NPC, sistema de combate por turnos, inventário com filtros, autoridade de servidor — as graphs se tornam redes densas de nós conectados por fios que não têm equivalente em nenhuma metáfora de engenharia de software. Rastrear um bug numa Blueprint graph grande é fundamentalmente mais difícil do que rastrear em código texto. C++ em Unreal não é o C++ padrão: a engine usa um sistema de reflexão próprio com macros `UCLASS`, `UPROPERTY`, `UFUNCTION`, um garbage collector customizado, convenções de herança que divergem do C++ moderno e um ciclo de compilação que para projetos grandes pode levar minutos. Para um engenheiro sênior confortável com Java, Kotlin ou Python, C++ Unreal representa uma curva de aprendizado substancialmente maior do que GDScript, GML ou Lua — e sem o ganho de um 2D nativo que justifique o investimento.

Na dimensão de multiplayer, Unreal tem uma stack server-authoritative genuinamente robusta — `AGameMode`, `APlayerController`, replicação por `UPROPERTY(Replicated)`, `RPC`s de servidor e cliente, `GameplayAbilitySystem` para efeitos e habilidades. Para jogos 3D competitivos com predição de client-side e reconciliação de estado, é a referência da indústria. Para um RPG 2D top-down com combate por turnos, inventário persistente e dois jogadores vendo o mesmo mapa — toda essa maquinaria existe, mas nenhuma parte dela é simplificada pelo fato do jogo ser 2D. Você aprende a replicação do Unreal, o GAS, o modelo de autoridade — ao mesmo tempo que lida com Paper2D e a infraestrutura 3D — sem nenhum atalho pelo recorte do projeto.

A licença do Unreal é gratuita com royalty de 5% sobre faturamento acima de $1 milhão — para um projeto pessoal sem pretensão comercial imediata, isso não é eliminatório. Mas é o único ponto que não penaliza Unreal neste crivo. Todas as demais dimensões que importam para o projeto — footprint, ergonomia de editor 2D, custo de aprendizado para o perfil do leitor, tempo até o primeiro frame jogável — ficam aquém das alternativas mais enxutas.

O padrão que emerge ao longo deste subcapítulo é que cada engine descartada falha por razões diferentes, e as razões importam. Unity falhou por escolhas históricas de arquitetura (3D-first) e modelo de negócio. Phaser falhou por ser framework, não engine, com multiplayer inteiramente fora do escopo. GameMaker, Defold e Construct falharam por paradigmas de rede inadequados ao projeto. Unreal falha por excesso: uma ferramenta construída para produção AAA 3D, com toda a complexidade que isso implica, aplicada a um RPG 2D top-down de projeto pessoal. É a diferença entre uma faca de chef e um torno mecânico — o torno é mais poderoso, mas para o que se precisa cortar, a faca é a ferramenta certa.

O próximo conceito fecha o subcapítulo com o mapa de trade-offs consolidado: todas as engines lado a lado pelas sete dimensões, mostrando o padrão de falhas e apontando o buraco que as alternativas deixam — o buraco que Godot 4 preenche.
